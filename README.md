## 6. 세 도구의 결과를 일치시키기 위해 맞춘 것

문법만 옮겨 적으면 세 도구의 결과는 **반드시 달라집니다.** 실제로 확인한 원인은 세 가지였고, 각각 이렇게 해결했습니다.

| 원인 | 무엇이 달랐나 | 해결 |
|---|---|---|
| **결측 그룹** | Pandas `groupby`는 결측(NaN) 그룹을 기본으로 빼버리는 반면, Polars·DuckDB는 하나의 그룹으로 남긴다. 결측 지역 1만 건·카테고리 8천 건이 통째로 갈렸다. | 세 도구 모두 `미상`/`미분류`로 채운 뒤 집계 |
| **분위수 보간** | Polars `quantile()` 기본값은 `nearest`, Pandas·DuckDB는 `linear`. 각자 계산하면 IQR 경계가 어긋나 걸러지는 행이 달라진다. | IQR 경계를 Pandas로 **한 번만** 계산해 세 도구에 같은 숫자로 전달 |
| **동점 정렬** | `total`만으로 정렬하면 값이 같은 행의 순서가 도구마다 달라진다. | 정렬 기준을 `total 내림차순 → region → category`까지 통일 |

추가로 Pandas는 `encoding="utf-8-sig"`를 지정해야 BOM 때문에 첫 컬럼명이 깨지지 않습니다. Polars·DuckDB는 자동으로 처리합니다.

## 7. 실행 결과 요약

100만 행 기준, `timeit number=10` (세 도구 동일, 예열 1회 후 측정, CSV 읽기 포함):

| 순위 | 도구 | 1회 평균 | Pandas 대비 |
|---:|---|---:|---:|
| 1 | Polars Lazy | 약 0.039초 | 약 14배 |
| 2 | DuckDB SQL | 약 0.078초 | 약 7배 |
| 3 | Pandas | 약 0.564초 | 기준 |

세 도구의 집계 결과는 **모두 동일**했습니다. 가장 최근 실행의 정확한 수치는 [`Docs/result.md`](Docs/result.md)에 있습니다.

### 반복 횟수를 10회로 정한 근거

가장 빠른 Polars는 1회가 0.04초로 짧아 잡음에 취약합니다. 20회 표본으로 'Pandas 대비 배수'의 흔들림을 재봤습니다.

| N | Polars 배수 흔들림 | 비고 |
|---:|---|---|
| 1 | ±20% | 결론이 뒤집힐 정도 |
| 3 | ±9% | 요구사항 최소치 |
| **10** | **±3.5%** | 채택 (총 측정 약 7초) |
| 20 | ±2% 수준 | 시간 2배, 개선폭 미미 |

또한 **예열이 반복 횟수보다 중요했습니다.** Polars의 첫 실행은 이후 실행보다 **약 68% 느립니다**(0.066초 → 0.039초). 파일 캐시가 비어 있고 엔진 초기화가 함께 일어나기 때문인데, 이는 집계 성능이 아닙니다. `number=3`이면 이 한 번이 측정값의 1/3을 오염시킵니다.

실제로 별도 프로세스로 3회 실행한 결과 배수는 13.71 / 14.43 / 15.20 (±5%)였습니다. 위 표의 ±3.5%보다 조금 넓은데, 표본이 한 세션 안에서 얻은 값이라 프로세스마다 달라지는 초기화 비용과 시스템 부하까지는 반영하지 못했기 때문입니다. `number=3`일 때의 10.59 / 13.23 / 13.91 (±14%)과 비교하면 개선은 분명합니다.
